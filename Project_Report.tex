\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}
\noindent
In today's digital era, booking event tickets seamlessly and securely is a growing challenge due to high user traffic, concurrent bookings leading to double-booking errors, and unoptimized mobile experiences. \textbf{InstantBook} is a comprehensive web application and Progressive Web App (PWA) designed to streamline the entire lifecycle of event ticket booking through real-time state synchronization, an intuitive seat selection UI, and secure payment processing.

InstantBook enables two major user roles—\textit{Administrator and User}—to operate within a unified environment. The platform allows administrators to create events, manage large seating layouts (such as interactive 15-row by 25-column seat maps), and easily verify ticket validity. Users can instantly browse available events, view live seat availability with granular "Locked" and "Booked" statuses, complete frictionless payments via Razorpay integration, and receive QR-coded digital tickets directly to their email and PWA dashboards. 

The system relies on a modern, robust technology stack. The frontend is built using \textbf{Next.js 16, React, and Tailwind CSS}, offering a highly responsive, dynamic, and native-like PWA user experience. The backend services utilize \textbf{Node.js, Express.js, PostgreSQL coupled with the Prisma ORM}, and \textbf{Upstash Redis} for handling high-concurrency real-time seat locking algorithms. Additional integrations include \textbf{Resend} for automated email notifications and \textbf{HTML5 QR Code} for rapid ticket verification by event staff.

InstantBook successfully addresses major ticketing pain points—eliminating double bookings through real-time distributed locking, improving mobile accessibility via PWA capabilities, and providing a premium, transparent booking experience. By combining modern modular architecture with seamless third-party gateways, this project demonstrates a highly scalable and resilient approach to digital event management and ticketing.

\chapter{Introduction}
\section{Introduction}
In the fast-paced modern world, the process of discovering events and booking tickets requires speed, reliability, and accessibility. However, traditional ticketing platforms often struggle with high traffic scenarios, confusing seat layouts, and concurrency issues where multiple users accidentally book the same seat. To overcome these challenges, \textbf{InstantBook} was specifically developed as an advanced event ticket booking platform that optimizes every stage of the user journey—from event discovery to digital ticket validation at the venue.

InstantBook is engineered to serve both event organizers (Administrators) and the general audience (Users). It enables administrators to effortlessly set up events, define venue capacities using an interactive seat map manager, and securely scan tickets. For the end-user, it delivers a smooth natively-installable Progressive Web App (PWA) experience, complete with an interactive live seat map, secure checkout via Razorpay, and QR code integration for purely digital, paperless entry.

\section{Motivation}
Event ticketing systems are notoriously difficult to scale effectively. One of the most prevalent challenges occurs during high-demand periods when hundreds of users view the exact same available seats simultaneously. Without an immediate real-time locking mechanism, users frequently experience checkout failures, or worse, double-booked scenarios requiring massive manual remediation. 

Additionally, users expect a fluid experience on their smartphones without needing to download large native applications from app stores. Traditional web apps lack the push notifications, offline caching, and native feel of installed apps.

The need to solve concurrency challenges transparently while delivering a premium mobile-first app experience strongly motivated the development of InstantBook. By utilizing in-memory databases like Redis for instantaneous seat locking, Next.js for high-performance server-side rendering, and PWA techniques for seamless installation, InstantBook aims to rival industry standards while remaining highly accessible and scalable.

\section{Problem Definition}
Currently, minor and mid-tier organizations face significant technical hurdles in ticketing:
\begin{itemize}
    \item \textbf{Concurrency conflicts:} Multiple users attempt to pay for the exact same seat concurrently, leading to failed transactions.
    \item \textbf{Complex seat verification:} Traditional ticketing requires either paper printouts or insecure emailed PDFs that are tedious to verify manually.
    \item \textbf{Poor mobile adaptation:} Users often interact with clunky responsive web designs that fail to mimic native app layouts, resulting in high drop-off rates.
    \item \textbf{Lack of transparency:} Seat maps often fail to differentiate between "Locked" (currently in someone's cart) and simply "Available" or "Booked", creating a frustrating guessing game for users.
\end{itemize}

InstantBook was directly designed to address and eliminate these exact infrastructural and user experience bottlenecks.

\section{Objective of the Project}
The underlying objective of \textbf{InstantBook} is to provide a robust, cloud-based platform that brings enterprise-level ticketing capabilities to a broader market while prioritizing speed and reliability. The key objectives include:
\begin{enumerate}
    \item Implementing a \textbf{real-time seat locking mechanism} utilizing Redis to entirely eliminate double-booking overlaps.
    \item Providing a highly interactive \textbf{15x25 grid seat map layout} using React, capable of instantly reflecting "Available", "Locked", and "Booked" statuses.
    \item Constructing the platform as a \textbf{Progressive Web App (PWA)} to provide easy installation, app-like interactions, and faster load times.
    \item Streamlining checkout through a secure \textbf{Razorpay API integration}.
    \item Equipping tickets with dynamically generated \textbf{QR logic} and building a robust validation gateway for fast venue check-ins.
    \item Establishing a modern backend backbone via \textbf{Express.js, PostgreSQL}, and the \textbf{Prisma ORM} for maximum structural integrity.
\end{enumerate}

\section{User Requirement Analysis}

InstantBook is conceptualized to support dynamic ticketing environments. To clearly define the scope and architecture, assessing the system through detailed \textbf{functional} and \textbf{non-functional} requirements is essential.

\subsection{Functional Requirements}

\subsubsection*{1. Administrator Requirements}
\begin{itemize}
    \item Admin role must execute secure portal access via JWT authentication.
    \item Interface to create new events, uploading banners via Cloudinary, and setting date/time logic.
    \item Interactive tools to configure seat availability limits and map capacities (e.g., standardizing a 15x25 array).

\subsubsection*{2. User (Audience) Requirements}
\begin{itemize}
    \item Users should register and authenticate properly, allowing secure history tracking.
    \item Browse global or local active events comprehensively.
    \item Select exact assigned seats utilizing a visual 2D interactive UI (e.g., matrix grid).

\chapter{System Analysis}

\section{Existing System}

Conventional standalone ticket booking frameworks often operate on rudimentary principles, creating a variety of technical pain points:
\begin{itemize}
    \item \textbf{Synchronous Database Checking:} Traditional systems check seat availability via direct relational database queries immediately prior to payment authorization. Under load, this delay allows two users to query an 'Available' seat almost simultaneously, resulting in a conflicting transaction.
    \item \textbf{Clunky Seat Arrays:} Older applications rely heavily on HTML select inputs or non-interactive static images, leaving users confused about their physical proximity to the stage or screen.
    \item \textbf{Disjointed Mobile Experience:} Due to budget constrictions, companies bypass native app development entirely, forcing users through standard mobile browsers which lack cache retention and direct notification support.
    \item \textbf{Verification Friction:} Paper tickets or barcode verification mandates expensive physical scanning hardware at event entries.
\end{itemize}

\section{Proposed System}

\textbf{InstantBook} comprehensively modernizes the entire workflow utilizing an edge-ready, scalable ecosystem. 

\subsection*{Features of the Proposed System}
\begin{itemize}
    \item \textbf{Real-Time 'Locked' Seat States:} Immediately upon engaging the checkout cart, user requests pass to Upstash Redis. Redis rapidly caches the seat as 'Locked' for a fixed timer, immediately preventing all other viewers from clicking it. 
    \item \textbf{Progressive Web App Architecture:} Built via Next.js manifested traits, the interface is readily installable to the device desktop, boasting offline caching capabilities for previously accessed e-tickets.
    \item \textbf{Client-Side Validation and Webhooks:} Once Razorpay approves the transaction, InstantBook directly transitions the 'Locked' seat metric to a deeply verified 'Booked' metric in the PostgreSQL main database, bridging client transparency and secure server logs.
    \item \textbf{Integrated QR Verification:} Harnessing the HTML5-QRCode module allows venue staff to securely validate complex encrypted ticket strings utilizing practically any standard smartphone camera.
\end{itemize}

\subsection*{Advantages of the Proposed System}
\begin{itemize}
    \item Entirely bypasses transaction-conflict issues via Redis distributed caching.
    \item Enhances mobile user retention with frictionless PWA onboarding.
    \item Visual, high-fidelity 15x25 interactive seat maps drastically improve the user curation process.
    \item Significantly decreases physical overhead required for venue check-ins.
\end{itemize}

\chapter{System Design}

System design details the structural layout mapping interactions between dynamic components. InstantBook operates under a decoupled Client-Server architecture wherein React elements maintain state synchronization with Express.js endpoints.

\section{UML Diagrams and Workflows}

Unified Modeling Language methodologies clarify relational behaviors inherent inside the platform.

\subsection*{1. Use Case Diagram Context}
The system dictates two primary boundaries:
\begin{itemize}
    \item \textbf{User Role:} Interfaces with Registration nodes, Seat Maps, the Razorpay Checkout flow, and Personal Ticket Dashboards.
    \item \textbf{Admin Role:} Modulates Event Creation (Banners, Timestamps, Capacities) and Scanning Interfaces (QR validation endpoints).
\end{itemize}

\subsection*{2. Interaction Flow (Booking Sequence)}
\begin{enumerate}
    \item The user initiates a grid map interaction and requests "Seat G14".
    \item The Next.js frontend deploys a highly concurrent ping to the Express API.
    \item The API queries the Redis memory partition; if clear, Redis locks "Seat G14" with a 5-minute Time-To-Live (TTL).
    \item The frontend updates immediately for all users globally rendering G14 in an amber 'Locked' hue.
    \item Following a positive verification webhook string from Razorpay, Express validates and definitively commits the row to PostgreSQL as 'Booked'. 
    \item An automated workflow executes the Resend API to deliver the QR attachment reliably.
\end{enumerate}

\section{Data Storage and Access Architecture}

\begin{itemize}
    \item \textbf{PostgreSQL + Prisma:} Maintains primary ACID-compliant persistence matrices corresponding to Users, Events, and confirmed Bookings. Prisma supplies rigid schema generation alongside automated GraphQL-like querying capabilities.
    \item \textbf{Upstash Redis:} A key-value architectural extension strictly deployed to manage transient 'Locked' state caching prior to confirmed sale.
    \item \textbf{Cloudinary:} Operates purely via cloud to intercept, format, and deliver all media assets uniformly.
\end{itemize}

\section{Description of Core Modules}

\subsection{Seat Configuration Module}
\textbf{Purpose:} Enables reliable interactivity matching the 15x25 array interface.
\begin{itemize}
    \item \textbf{Frontend UI:} Nested grid iterations evaluate each seat via mapping algorithms matching real-time API polling data.
    \item \textbf{Data Handlers:} Differentiates actively between empty coordinates, deeply booked, and currently locked states, applying distinctive CSS class structures instantly (using Tailwind-Merge).
\end{itemize}

\subsection{Checkout and Locking Module}
\textbf{Purpose:} Guarantee transactional integrity leveraging Upstash Redis caches.
\begin{itemize}
    \item An API endpoint assigns a temporary unique identifier via the Redis key mechanism preventing collisions when heavy multi-user traffic filters into the cart initialization sequence.
    \item Integrates Razorpay script tags directly ensuring Payment Card Industry (PCI) compliance. The frontend bypass logic resolves seamlessly via client verifications specifically calibrated to avoid blocking execution pathways.
\end{itemize}

\subsection{PWA and Validation Gateway}
\textbf{Purpose:} Bridge native features and hardware scanning interfaces.
\begin{itemize}
    \item A comprehensive Service Worker securely caches external JS chunks and CSS layouts rendering the shell instantly visible even during heavy network drops.
    \item The QR engine ingests stringified JSON hashes encoding metadata which HTML5-QRCode scripts efficiently unpack dynamically directly from mobile device lens sensors at the threshold point. 
\end{itemize}

\chapter{Testing}

\section{Security Paradigms}
\begin{itemize}
    \item JWT signatures explicitly mandate secure header transmissions avoiding localized cookie interception scenarios.
    \item Prisma inherently parameterizes incoming DB queries aggressively halting traditional SQL Injection attempts mechanically.
    \item Razorpay transactions ensure the backend code never materially processes raw credit ledger logic, relying directly on verified transaction hashes dynamically.
\end{itemize}
